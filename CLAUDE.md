## Project Overview

**Nexus** — Role-Based Event Operations Platform. A full-stack Next.js 16 app for managing event registrations, check-ins, and meal distribution via QR codes.

## Commands

```bash
npm run dev       # Start dev server (Next.js)
npm run build     # Production build
npm run start     # Start production server
npm run lint      # Run ESLint
```

No test suite is configured.

## Architecture

### Stack
- **Framework:** Next.js 16 App Router with React 19 and TypeScript
- **Database/Auth:** Supabase (PostgreSQL + Supabase Auth, no ORM)
- **Styling:** Tailwind CSS 4 with glassmorphism/dark theme
- **Email:** Nodemailer via Gmail SMTP (`lib/mailer.ts`)
- **QR:** `html5-qrcode` for scanning, `react-qr-code` for generation

### Key Architectural Decisions

**Supabase clients are split by privilege:**
- `lib/supabase.ts` — public browser-safe client (`NEXT_PUBLIC_*` keys)
- `lib/supabase-admin.ts` — server-side only, bypasses RLS (uses `SUPABASE_SERVICE_ROLE_KEY`). Never import this in client components.

**Registration uses a DB-level atomic RPC** (`register_team_with_capacity_check`) to prevent race conditions when checking track capacity. The API route at `app/api/register/route.ts` calls this RPC, then rolls back on failure.

**QR tokens are time-limited (30-second expiry)**, generated by `app/api/generate-qr/route.ts`. The volunteer scanner (`components/volunteer/QRScanner.tsx`) has a 2-second debounce to prevent duplicate scans.

### Roles & Routing
- **Candidate** → `app/candidate/dashboard/page.tsx` (view team status, generate QR)
- **Volunteer** → `app/volunteer/dashboard/page.tsx` (scan QR for check-in/meals, live stats)
- **Admin** → `app/admin/dashboard/page.tsx` (approve/reject registrations)

### Email Flow
Three email states handled in `lib/mailer.ts`:
1. `sendPendingRegistrationEmail` — on registration submission (payment pending)
2. `sendCredentialEmail` — on admin approval (sends team credentials/password)
3. `sendRejectionEmail` — on admin rejection

### API Routes
- `POST /api/register` — team registration (validates SRN/email uniqueness, calls RPC, sends email). Public, no auth required.
- `POST /api/generate-qr` — generate 30-sec QR token. Requires `Authorization: Bearer <token>` header; verifies caller owns the team.
- `POST /api/admin/approve` — approve registration + send credentials. Requires admin role via Bearer token.
- `POST /api/admin/reject` — reject registration + notify team. Requires admin role via Bearer token.

**Auth pattern for protected API routes:** clients call `supabase.auth.getSession()` to get the access token and include it as `Authorization: Bearer <token>`. Server-side routes call `supabaseAdmin.auth.getUser(token)` to verify.

### Security Notes
- QR tokens and passwords use `crypto.randomBytes` (Node built-in) — never `Math.random()`.
- All user-supplied strings rendered in emails go through `escapeHtml()` in `lib/mailer.ts`.
- Security headers (X-Frame-Options, X-Content-Type-Options, etc.) are set in `next.config.ts`.
- Volunteer dashboard uses Supabase realtime (not polling) for live venue updates.

### Environment Variables
```
NEXT_PUBLIC_SUPABASE_URL          # Supabase project URL
NEXT_PUBLIC_SUPABASE_ANON_KEY     # Public anon key
SUPABASE_SERVICE_ROLE_KEY         # Admin key (server only)
SUPABASE_DATABASE_PASSWORD        # DB password
EMAIL_USER                        # Gmail address for Nodemailer
EMAIL_PASS                        # Gmail app password
```

### Path Alias
`@/*` maps to the project root (e.g., `@/lib/supabase`, `@/components/auth/LoginModule`).
